# Keep Markdown rendering UI-independent

The Markdown rendering module owns each rendering mode's content semantics, including a closed catalog of embedded elements and their properties, but it does not own UI-framework components or lifecycle. A destination must exhaustively realize the elements its selected mode can emit and chooses whether to replace, prerender, server-render, hydrate, or otherwise make them interactive; missing support fails fast instead of leaving inert output. This seam deliberately exposes the simple HAST representation to destinations that render UI elements instead of introducing another adapter abstraction or leaking UI-framework knowledge into the Markdown module. This preserves use by both backend and frontend callers while making today's custom-element coordination explicit.
